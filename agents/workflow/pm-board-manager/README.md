# PM Board Manager

## Overview

일일/주간업무보고와 **이전 PM 보드**를 시간축으로 교차 분석하여, 사용자(실무자)를 관리해주는 **개인 PM** 역할을 수행하는 에이전트입니다. 무엇이 밀렸고(지연·정체), 무엇이 막혔으며(리스크·블로커·의존성), 무엇을 먼저 해야 하는지(우선순위·계획), 여러 프로젝트가 지금 어떤 상태인지(포트폴리오)를 추적하여 **"살아있는 PM 보드"**(`PM보드/PM-Board.md`) 하나를 유지·갱신합니다.

기존 보고 에이전트(일일/주간 작성, 팀장 브리핑, PPT)가 "과거 실적을 위로 보고"하는 백워드 흐름인 반면, 이 에이전트는 **미래·횡단 관리**라는 PM 고유 영역을 담당합니다.

## Category

`workflow` - 개인 프로젝트 관리(PM) 자동화 워크플로우

## Prerequisites

- Obsidian 앱 설치 및 실행
- [Obsidian Local REST API](https://github.com/coddingtonbear/obsidian-local-rest-api) 플러그인 설치 및 활성화
- Claude Code MCP 설정에 obsidian 서버 등록
- `일일업무보고/` 디렉토리에 `YYYY-MM-DD.md` 형식의 일일보고서 존재
- Python 3 설치 (영업일 계산용)

## 핵심 특징: 살아있는 보드 (시간축 추적)

이 에이전트의 가치는 **이전 보드와의 비교**에서 나옵니다. 단순 스냅샷이 아니라, 매 실행 시 이전 보드를 기준점으로 삼아 변화를 누적 계산합니다.

| 추적 항목 | 의미 |
|---|---|
| **약속 이행률** | 며칠 전 "내일 할 일"로 적은 것이 실제로 끝났는가 |
| **리스크 경과일** | 블로커가 며칠째 열려 있는가 (에이징) |
| **정체 일수** | 같은 업무의 진행률이 며칠째 멈췄는가 |

## Capabilities

- 이전 PM 보드 + 일일/주간보고의 시간축 교차 분석
- 약속 이행 추적 및 이행률 산정 (이행/부분이행/미이행)
- 진행률 정체 감지 (기본 3영업일 이상 변동 없음)
- 리스크 에이징 — 최초 발견일 보존, 경과일 누적, 상태 기계(신규/진행중/악화/해결)
- 외부 의존성 대기 추적
- WIP 모니터링 및 과다 경고(기본 임계 8) + 미룰 후보 제안
- 다음 액션 우선순위(P0~P2) 배정
- 멀티 프로젝트 포트폴리오 보드 + 마일스톤 추적
- 단일 보드 파일 유지 (현황 교체 + 이력 누적)

## Usage

### Basic Usage

```
PM 점검해줘
```

에이전트가 자동으로:
1. 최근 영업일~오늘 범위 계산
2. `PM보드/PM-Board.md` 읽기 → 초기/갱신 모드 판별
3. 일일업무보고 수집 + 이전 보드와 시간축 비교
4. 약속 이행·정체·리스크 에이징·WIP·포트폴리오 분석
5. 보드 초안 제시 후 저장

### 기간 지정

```
최근 2주 PM 보드 갱신해줘
```

### 설정 조정

보드 frontmatter의 `config`로 임계값을 조정할 수 있습니다:

```yaml
config: { 정체임계일: 3, WIP임계: 8 }
```

### 마일스톤 직접 정의

보드 frontmatter의 `milestones`에 프로젝트별 마일스톤을 적어두면 추론보다 우선 적용됩니다:

```yaml
milestones: { "ATL-RAMS ALE": "i18n 사이클 종료", "lynkit_app": "M2 Alpha 빌드" }
```

## Best Practices

1. **주기적으로 호출하세요**: 매주 또는 격주로 호출해야 시간축 추적(경과일·정체 일수)이 의미를 가집니다
2. **약속을 일일보고에 기록하세요**: `## 내일 할 일`이 충실할수록 약속 이행률 추적이 정확해집니다
3. **마일스톤을 직접 정의하세요**: frontmatter `milestones`에 적어두면 포트폴리오 추적 품질이 올라갑니다
4. **임계값을 본인 패턴에 맞추세요**: 처리량이 많으면 `WIP임계`를 올리세요 (기본 8)
5. **초안을 꼭 확인하세요**: 약속 이행 매칭은 오탐 가능성이 있어 저장 전 검토가 중요합니다

## Limitations

| 가능 | 불가능 |
|---|---|
| 일일/주간보고 + 이전 보드 기반 PM 분석 | 보고서 없이 자동 관리 |
| 약속 이행률·경과일·정체 일수 누적 추적 | 보고서에 없는 업무 자동 예측 |
| 리스크 에이징 및 상태 추적 | 외부 일정 시스템(JIRA 등) 직접 연동 |
| 멀티 프로젝트 포트폴리오 조망 | 다른 사람의 보고서 관리 |
| 단일 보드 유지(현황 교체+이력 누적) | Notion, Google Docs 등 타 서비스 |

## Troubleshooting

### "Obsidian 볼트에 연결할 수 없습니다"
- Obsidian 앱 실행 및 Local REST API 플러그인 활성화 확인

### 경과일/정체 일수가 0으로만 나올 때
- 첫 실행(이전 보드 없음)이거나 이전 보드 파싱 실패일 수 있습니다
- 두 번째 실행부터 누적 추적이 정상 동작합니다

### 약속 이행 매칭이 부정확할 때
- 일일보고의 "내일 할 일"과 이후 "오늘 한 일"의 제목/괄호 기술용어가 일치할수록 정확합니다
- 초안 확인 시 매칭을 수정 요청할 수 있습니다

### 변경 이력이 사라질 때
- 갱신 시 이력 보존이 핵심입니다 — 초안에서 `## 5. 변경 이력`이 이전 내용을 포함하는지 확인하세요

## Common Use Cases

1. **주간 자기 점검**: 한 주를 마치며 밀린 것·막힌 것·다음 우선순위 정리
2. **멀티 프로젝트 조망**: 여러 프로젝트를 동시 진행할 때 한 보드로 상태 파악
3. **약속 이행 추적**: 스스로 한 약속이 실제로 지켜지는지 누적 모니터링
4. **리스크 관리**: 블로커가 방치되지 않도록 경과일로 압박
