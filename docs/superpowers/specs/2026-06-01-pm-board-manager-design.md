# PM 보드 관리자 (pm-board-manager) 설계 문서

- 작성일: 2026-06-01
- 상태: 설계 확정 (구현 대기)
- 방식: B (상태 누적형 "살아있는 PM 보드")

## 1. 배경 및 목적

옵시디언 업무보고 관련 기존 에이전트는 모두 **과거 실적을 위로 보고하는 백워드 흐름**이다:

| 에이전트 | 역할 | 방향 |
|---|---|---|
| daily-work-reporter | 일일보고 작성 | 기록 |
| weekly-work-reporter | 주간보고 집계·차트 | 기록 요약 |
| team-lead-weekly-briefing | 임원 구두 브리핑 | 상향 보고 |
| weekly-report-ppt-* | 보고용 PPT | 상향 보고 |

**미래를 내다보고 프로젝트를 가로질러 관리하는 PM 고유 영역**이 비어 있다. 본 에이전트는
사용자(실무자)를 관리해주는 **개인 PM** 역할을 수행한다 — 지연·리스크를 짚어주고,
우선순위와 계획을 잡아주며, 약속 이행을 추적한다.

보고 데이터에는 PM이 일할 재료가 충분하다: 진행률(`30→80%`), "내일 할 일"(=약속),
리스크/특이사항, 외부 협의 대기, 프로젝트별 공수, 일 처리량 추이.

## 2. 책임 범위 (사용자 확정)

1. 지연·정체 탐지
2. 리스크·블로커·의존성 추적
3. 우선순위·계획 수립
4. 마일스톤·포트폴리오 보드

결과물 형태: **옵시디언 파일** (기존 방식 계승).

## 3. 에이전트 정체성 · 입출력

- **에이전트명:** `pm-board-manager`
- **위치:** `agents/workflow/pm-board-manager/` (`pm-board-manager.md` + `agent.json` + `README.md` + `examples/`)
- **역할 한 줄:** 일일/주간보고를 읽어 살아있는 PM 보드 파일 하나를 유지하며,
  실행할 때마다 이전 보드와 비교해 지연·리스크·계획·포트폴리오를 갱신하는 내 개인 PM.

### 입력 (모두 읽기 전용)

- 트리거: `"PM 점검해줘"`, `"PM 보드 갱신해줘"`, 기간 지정(`"이번 주"`, `"최근 2주"`)
- 소스:
  - `일일업무보고/*.md` — 분석 기간(기본: 최근 영업일 ~ 오늘)
  - `주간업무보고/*.md` — 맥락 보강용(선택)
  - **이전 PM 보드** `PM보드/PM-Board.md` — 시간축 비교의 핵심 입력
- 날짜 계산: weekly-work-reporter의 Python 원라이너 패턴 재사용 (영업일 기준)

### 출력 (이 에이전트가 쓰는 유일한 파일)

- `PM보드/PM-Board.md` — 단일 살아있는 보드
  - 현황 섹션(0~4): 매 실행 시 최신값으로 교체(patch)
  - 변경 이력(섹션 5): 누적(append)

### 기존 에이전트와의 경계

- weekly-work-reporter의 차트(공수/유형 pie)를 재생성하지 않고 입력으로 참고만 함.
- team-lead-weekly-briefing은 임원 구두 보고용, 본 보드는 본인 관리용 내부 도구.
- 공통 원칙 계승: 원본 보고서 절대 수정 금지, 저장 전 초안 확인, 명사형 종결, 비전공자 가독성.

## 4. PM 보드 파일 포맷

```
---
type: PM보드
updated: YYYY-MM-DD
period_analyzed: YYYY-MM-DD ~ YYYY-MM-DD
sources_daily: [2026-05-20, ...]
config: { 정체임계일: 3, WIP임계: 5 }
tags: [pm-board, management]
---

# PM 보드 — 갱신 YYYY-MM-DD

## 0. 한눈에 (PM 총평)
- 약속 이행률 N/M (XX%) · 정체 N건 · 열린 리스크 N건 · WIP N건
- [PM 한 줄 코멘트 — 이번 점검에서 가장 주목할 신호]

## 1. 지연·정체 (지금 봐야 할 것)
### 미이행 약속
| 업무 | 약속일 | 경과(영업일) | 현재 상태 | 비고 |
|---|---|---|---|---|

### 정체 업무 (진행률 N일째 정지)
| 업무 | 프로젝트 | 진행률 | 정체 일수 | 마지막 변동일 |
|---|---|---|---|---|

## 2. 리스크·블로커·의존성
| 리스크/블로커 | 프로젝트 | 최초 발견 | 경과일 | 상태 | 대응 |
|---|---|---|---|---|---|

### 외부 의존성 대기
| 대기 항목 | 대기 대상 | 대기 일수 |
|---|---|---|

## 3. 다음 액션 (우선순위)
1. **[P0] [업무]** — [왜 지금/근거 한 줄]
2. **[P1] ...**

### 미룰 것 / WIP 경고
- [WIP 임계 초과 시 경고 + 미룰 후보]

## 4. 프로젝트 포트폴리오 보드
| 프로젝트 | 상태 | 최근 활동일 | 진행 중 | 열린 리스크 | 다음 마일스톤 |
|---|---|---|---|---|---|

## 5. 변경 이력 (Changelog)
### YYYY-MM-DD 갱신
- 신규 리스크 N · 해결 N · 약속 이행 N/M · 신규 정체 N
- [이전 보드 대비 핵심 변화 1~2줄]
```

### 포맷 설계 의도

- 섹션 0이 PM 대시보드 요약 — 이것만 봐도 상태 파악.
- 섹션 1·2의 시간축 컬럼(경과일/정체 일수)이 방식 B의 가치를 드러냄.
- 섹션 4 마일스톤: 보고서에 명시 데이터가 없으므로 **추론 + 사용자 직접 입력 허용**
  (보드에 마일스톤을 적어두면 그대로 추적). 사용자 입력 마일스톤이 추론보다 우선.
- 섹션 5는 누적 → 한 파일에서 주간 변화 흐름 추적.

## 5. 분석 로직 (방식 B의 핵심)

매 실행 시 4단계로 계산하며, 핵심은 모두 **이전 보드 + 신규 보고**의 비교다.

### ① 약속 이행 추적

- 약속 소스: 일일보고 `## 내일 할 일`, 주간보고 `## 다음 주 계획`, 이전 보드 "다음 액션".
- 대조: 이후 일일보고 `## 오늘 한 일`(완료)·`## 진행 중인 일`과 제목/괄호 기술용어로 매칭
  (weekly-work-reporter의 "동일 업무 판별" 규칙 재사용).
- 판정: `이행`(완료 등장) / `부분이행`(진행 중 등장) / `미이행`(N영업일 경과 무언급).
  미이행은 섹션 1에 노출.
- 산출: 약속 이행률 = 이행 ÷ 전체 약속.

### ② 정체 감지

- 동일 업무를 날짜축으로 추적해 진행률 변화 관찰.
- 진행률 변동 없음이 **정체임계일(기본 3영업일)** 이상 지속 && 진행률 < 100% → `정체` 플래그
  (마지막 변동일·정체 일수 기록).

### ③ 리스크 에이징 (상태 기계)

- 소스: 일일 `## 특이사항`, 주간 `## 리스크 및 이슈`, 이전 보드 열린 리스크.
- 이전 보드 리스크와 매칭하여 **최초 발견일 보존 → 경과일 계산**.
- 상태: `신규` / `진행중` / `악화`(경과일↑ 무진전) / `해결`(이번 기간 해소 확인 시
  Changelog로 내리고 보드 본문에서 제거).
- 의존성: "외부 협의 대기 / 팀 협의 필요 / 결정 대기" 키워드 → 의존성 대기 목록(대기 일수 포함).

### ④ 우선순위·계획

- WIP 카운트: 현재 동시 "진행 중" 업무 수 → **WIP임계(기본 5)** 초과 시 경고 + 미룰 후보 제안.
- 다음 액션 우선순위 배정(P0→P2):
  - **P0**: 악화된 리스크 대응, 며칠째 미이행 약속, 외부 의존성 해소 액션
  - **P1**: 정체 업무 재가동, 마일스톤 임박 항목
  - **P2**: 신규 착수 예정, 일반 계획 항목

### 설정값

보드 frontmatter `config`에 두어 사용자가 조정 가능(정체임계일·WIP임계). 미지정 시 기본값
(정체임계일 3, WIP임계 5).

## 6. 엣지 케이스 · 안정성

- **첫 실행(이전 보드 없음):** 모든 항목 `신규`로 베이스라인 생성, 경과일·정체일수 = 0.
  Changelog에 "최초 생성" 기록.
- **멱등성(같은 기간 재실행):** 같은 날짜 Changelog 항목이 있으면 누적이 아니라 교체.
  현황 섹션(0~4)은 항상 최신값으로 patch.
- **부분 데이터:** 일부 날짜만 있어도 진행, 누락일 보드에 명시(weekly 패턴 재사용).
- **Obsidian 연결 실패:** 보드 내용 텍스트 출력 + Local REST API 확인 안내(기존 폴백 계승).
- **저장:** patch 시 H1::H2 헤딩 경로 사용(weekly에서 검증), 실패 시 append 폴백.
- **저장 전 초안 확인:** 항상 보드 전체를 보여주고 사용자 승인 후 저장.

## 8. 성공 기준

- 지정 기간의 일일보고 + 이전 PM 보드를 읽어 4개 영역(지연/리스크/계획/포트폴리오)을 산출.
- 약속 이행률·리스크 경과일·정체 일수가 이전 보드 대비 정확히 계산됨.
- 보드가 단일 파일로 유지되고, 현황은 교체·이력은 누적됨.
- 첫 실행/부분 데이터/연결 실패 등 엣지 케이스가 폴백으로 처리됨.
- 저장 전 사용자 확인, 원본 보고서 무수정.
